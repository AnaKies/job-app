# Технический разбор архитектуры CeeVee: Справочник инженера

Этот документ представляет собой глубокое погружение в архитектуру системы CeeVee. Ориентирован на разработчиков и студентов технических специальностей, понимающих основы ООП, баз данных и сетевых протоколов.

---

## 1. Системная топология и Стек (High-Level)

CeeVee — это современное веб-приложение, разделенное на три физических и логических слоя:

1.  **Frontend (UI Runtime):** **Next.js (App Router)** на TypeScript. Занимается рендерингом, управлением состоянием на стороне клиента и взаимодействием с Backend через типизированный API.
2.  **Backend (Logic Runtime):** Отдельный сервис на **Node.js**. Почему не Next.js API Routes? Для долгоживущих процессов (скрейпинг вакансий, тяжелые вычисления ИИ) и поддержки протокола **MCP**, который требует стабильного жизненного цикла процесса.
3.  **Storage Layer:** **Supabase**. Используется не только как реляционная БД (**PostgreSQL**), но и как векторное хранилище (**pgvector**) для семантического поиска.

---

## 2. Архитектурный паттерн: Hexagonal (Ports & Adapters)

Мы отошли от классической многослойной архитектуры (Layered) в пользу **Гексагональной**. Это критично для проектов, где внешние API (LLM, сайты вакансий) часто меняются.

### Как это устроено:
*   **Domain Core (Центр):** Здесь лежат чистые сущности (`Resume`, `JobListing`) и бизнес-правила (логика мэтчинга). Ядро **не импортирует** ничего из внешних библиотек (ни Prisma, ни Axios).
*   **Ports (Интерфейсы):** Определяют, *что* ядру нужно от внешнего мира (например, `ICompanyDiscoveryPort`).
*   **Adapters (Реализация):** Внешние модули, которые реализуют порты. Например, `OpenAICompanyDiscoveryAdapter` или `CrunchbaseAdapter`.

**Зачем это студенту?** Это реализация принципа **D (Dependency Inversion)** из SOLID. Мы зависим от абстракций, а не от деталей реализации.

---

## 3. Организация кода: Monorepo

Проект структурирован как монорепозиторий (с использованием TurboRepo или npm workspaces):

*   `apps/web`: Веб-интерфейс.
*   `apps/api`: Бэкенд и MCP-сервер.
*   `packages/domain`: Общая бизнес-логика и определения портов.
*   `packages/shared`: Контракты данных (DTO), схемы валидации и общие утилиты. Это позволяет избежать дублирования типов между фронтендом и бэкендом.

---

## 4. Модель данных и RAG (Retrieval-Augmented Generation)

Мы используем гибридный подход к данным.

### Реляционная часть (PostgreSQL):
Таблицы `resumes`, `companies`, `job_listings` связаны классическими Foreign Keys. Это обеспечивает ссылочную целостность для трекинга откликов.

### Семантическая часть (pgvector):
Для реализации «умного» поиска мы используем эмбеддинги. 
*   **Процесс:** Резюме и описания вакансий разбиваются на чанки (отрывки), которые прогоняются через модель (например, `text-embedding-3-small`). На выходе — вектор (массив чисел) из 1536 измерений.
*   **Поиск:** Когда мы ищем работу, мы сравниваем вектор поискового запроса с векторами вакансий в БД, используя **косинусное сходство**. Это позволяет находить работу по смыслу («разработчик интерфейсов» ≈ «frontend engineer»), а не по ключевым словам.

---

## 5. Интерфейсы: REST und MCP

Система предоставляет два способа взаимодействия:

1.  **Task-oriented REST API:** Для фронтенда. Каждый эндпоинт решает конкретную задачу пользователя (например, `/api/match`).
2.  **MCP (Model Context Protocol):** Это инновационная часть. Мы экспонируем возможности бэкенда как инструменты для других ИИ-агентов. Если вы используете Claude или ChatGPT Desktop, они могут «подключиться» к CeeVee через этот протокол и использовать функции `scrape_career_page` или `match_resume` как свои собственные навыки.

---

## 6. Логика Скрейпинга (Scraping Engine)

Скрейпинг — самая нестабильная часть. Мы решаем это через **ATS-awareness**:
*   Система сначала детектирует провайдера (Greenhouse, Lever, Workday).
*   Для каждого провайдера есть свой адаптер, который знает, в каких HTML-тегах лежат данные.
*   Процессы скрейпинга разделены на **Синхронные** (быстрый поиск 1-2 страниц) и **Асинхронные** (обход десятков сайтов в фоновом режиме через очередь задач).

---

## 7. Оптимизация через Unified Type Safety (Моя рекомендация)

Для проекта на TypeScript самой большой проблемой является рассинхронизация типов. Я внедряю библиотеку **Zod**.

### Как это работает:
1.  Мы описываем структуру данных один раз в `packages/shared` как Zod-схему.
2.  Из этой схемы автоматически выводится TypeScript-тип.
3.  На бэкенде эта схема проверяет входящий JSON (Runtime validation).
4.  На фронтенде эта же схема проверяет данные форм.
5.  Для MCP эта схема автоматически генерирует JSON-Schema описание аргументов функции.

**Результат:** 100% гарантия того, что данные, которые ожидает бэкенд, именно в таком виде придут с фронтенда или от ИИ-агента.

*   **БД:** Гибридная (SQL + Vector Search).
*   **Инновация:** Поддержка протокола MCP для кросс-агентного взаимодействия.
*   **Надежность:** Сквозная типизация через Zod.
